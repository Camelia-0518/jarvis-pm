# Jarvis PM - 产品管理效率工具

## 项目汇报材料（务实版）

> **一句话定位**：面向产品经理的文档生成与项目管理效率工具，聚焦 PRD 撰写、评审准备和知识沉淀的自动化。

---

## 一、项目概述

### 1.1 我们做了什么

Jarvis PM 是一款**面向产品经理的内部效率工具**，通过结构化模板、自动化生成和知识库集成，解决产品需求文档（PRD）撰写耗时、评审材料准备繁琐、历史经验难以复用等痛点。

它不是替代产品经理做决策，而是**把重复性、格式化的工作自动化**，让 PM 把精力集中在需求分析和方案设计上。

### 1.2 核心价值（量化）

| 工作流 | 传统方式 | Jarvis PM | 效率提升 |
|:-------|:---------|:----------|:---------|
| PRD 撰写 | 2-3 天 | 10-30 分钟 | **80-90%** |
| 评审准备 | 1 天 | 5-10 分钟 | **90%** |
| 站会报告 | 15 分钟 | 1-2 分钟 | **90%** |
| 竞品分析 | 4-8 小时 | 10-15 分钟 | **85%** |

## 二、核心功能详解

### 2.1 智能 PRD 生成器

**功能描述**：
- 输入产品需求和目标用户，系统自动生成标准 8 章 PRD 结构
- 支持医疗、金融、电商、教育等行业自动识别和模板匹配
- **医疗行业内置合规检查点**：等保三级要求、患者隐私保护、数据安全审计、跨院区同步规范
- 输出支持 Markdown / JSON / 飞书文档 / GitHub Issue 格式导出
- 与 Obsidian 个人知识库集成，生成文档自动归档到指定目录

**技术说明**：
- 基于大语言模型（Kimi K2.5）进行内容生成
- 非简单模板填空，而是根据输入语义进行结构化和专业化扩展
- 已内置"病理切片借阅平台"、"医疗管理后台"等行业模板

### 2.2 多角色协作模拟系统

**功能描述**：
- 系统内置 5 个专业角色视角进行内容校验：
  - **产品战略**：需求完整性和目标对齐
  - **体验设计**：UI/UX 合理性和用户流程
  - **技术评估**：架构可行性和实现风险
  - **质量保障**：测试覆盖和安全合规
  - **协调者**：冲突识别和优先级排序
- 每个角色输出独立的检查意见和置信度评分
- 当不同角色出现分歧时，系统自动标记并提示人工确认

**实际价值**：
在正式评审前，先用系统做一次"预审"，提前发现文档漏洞和逻辑冲突，减少正式评审中的返工。

### 2.3 可视化工作流编排

**功能描述**：
- 基于 React Flow 的流程画布
- 预置 3 套标准工作流：
  - "从 0 开发新功能"（头脑风暴 → PRD → 评审准备）
  - "安全审查流程"（风险识别 → 合规检查 → 报告生成）
  - "PRD 完整输出包"（PRD + 评审材料 + 站会模板）
- 每个节点实时显示执行状态和进度

### 2.4 文档协作与评审支持

**功能描述**：
- PRD 编辑器支持 Markdown 编辑和实时预览
- 多人编辑时光标同步显示，段落级锁定避免冲突
- 评论系统支持 @提及、回复和解决状态跟踪
- 评审材料自动生成：会议议程、Q&A 预案、风险清单、关键决策点

### 2.5 需求到原型的快速落地

**功能描述**：
- 从 PRD 内容中提取 UI 需求，生成 HTML + Tailwind CSS 低保真原型
- 自动生成 OpenAPI 3.0 接口规范和 FastAPI CRUD 代码骨架
- 识别 UI 组件需求，生成 React + TypeScript 组件代码
- 支持 HTML / ZIP / JSON 导出

**实际价值**：
缩短需求到开发的传递链路，减少因文档理解偏差导致的返工。

### 2.6 个人知识库检索

**功能描述**：
- 对接本地 Obsidian 知识库
- 在生成文档或回答问题时，自动检索历史项目文档和经验沉淀
- 支持语义检索，提升历史资料复用率

## 三、技术架构

### 3.1 技术栈

| 层级 | 技术 | 说明 |
|:-----|:-----|:-----|
| 前端 | Next.js 15 + React 19 + TypeScript | 现代前端主流方案 |
| 样式 | Tailwind CSS 4 | 原子化 CSS，开发效率高 |
| 状态管理 | Zustand | 轻量状态管理 |
| 工作流画布 | React Flow | 开源流程图组件，成熟稳定 |
| 后端 | FastAPI + Python 3.11 | 异步高性能，自动生成 API 文档 |
| 数据库 | PostgreSQL + Redis | 关系型数据 + 缓存 |
| 内容生成 | Kimi / Claude / OpenAI | 多 Provider 配置，自动降级 |
| 部署 | Vercel + Railway | 现代化部署方案 |

### 3.2 工程质量

- **后端单元测试**：301 个测试用例全部通过
- **E2E 全流程测试**：31 个端到端测试全部通过
- **测试代码量**：约 4,786 行
- **编译状态**：前端 `next build` 一次编译成功
- **接口状态**：内容生成接口已接入真实模型（Kimi），非模拟数据

### 3.3 设计特点

1. **多模型容错**：主用 Kimi，异常时自动切换到 Claude 或 OpenAI
2. **认证模式灵活**：支持本地免登录单用户模式，也保留 JWT 认证扩展接口
3. **标准化 API**：统一返回结构，降低前后端对接成本
4. **技能插件化**：新功能以独立技能模块注册，核心代码不受侵入
5. **Obsidian 双向集成**：既是输出端（保存文档），也是输入端（检索知识库）

## 四、与现有竞品的差异

### 4.1 市场格局

| 产品 | 定位 | 主要问题 |
|:-----|:-----|:---------|
| Notion | 通用工作空间 | 国内访问不稳定，PRD 管理需自建模板 |
| 飞书项目 | 企业项目管理 | 对非飞书用户门槛高，缺乏 PRD 自动生成功能 |
| Jira | 研发任务管理 | 学习成本高，产品经理使用体验差 |
| 语雀 | 知识库文档 | 无自动化生成能力，纯手动撰写 |

### 4.2 Jarvis PM 的差异点

| 维度 | 现有工具 | Jarvis PM |
|:-----|:-----|:----------|
| **PRD 生成** | 手动撰写或复制模板 | **输入需求后自动生成标准结构** |
| **行业适配** | 通用，无行业增强 | **医疗行业内置合规检查点和术语库** |
| **知识沉淀** | 分散存储，检索困难 | **与 Obsidian 集成，自动归档和语义检索** |
| **需求到原型** | 完全断链 | **PRD 可直接导出原型和接口代码** |
| **评审辅助** | 人工准备材料 | **自动生成议程、风险清单和 Q&A 预案** |
