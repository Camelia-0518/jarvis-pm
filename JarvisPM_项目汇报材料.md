# Jarvis PM - AI 驱动的产品管理协作平台

## 项目汇报材料

> **一句话定位**：用对话方式 10 分钟完成原本需要 2 天的 PRD 撰写和评审准备。

---

## 一、项目概述

### 1.1 我们做了什么

Jarvis PM 是一个**专为产品经理打造的 AI 原生工作平台**，通过深度集成大语言模型（Kimi/Claude/OpenAI），实现从需求输入到 PRD 生成、从评审准备到站会报告的全流程 AI 辅助。

它不是传统工具的"AI 附加功能"，而是**从底层围绕 AI 能力重新设计**的产品管理工作流系统。

### 1.2 核心价值（量化）

| 工作流 | 传统方式 | Jarvis PM | 效率提升 |
|:-------|:---------|:----------|:---------|
| PRD 撰写 | 2-3 天 | 10 分钟 | **90%** |
| 评审准备 | 1 天 | 5 分钟 | **95%** |
| 站会报告 | 15 分钟 | 1 分钟 | **93%** |
| 竞品分析 | 4-8 小时 | 3 分钟 | **95%** |

## 二、核心功能详解

### 2.1 AI PRD 生成器（核心卖点）

**功能描述**：
- 通过多轮对话收集需求，AI 自动生成标准 8 章 PRD 结构
- 支持医疗、金融、电商、教育等行业自动检测和模板匹配
- 医疗行业自动增强：等保三级合规、患者隐私保护、医疗数据安全、操作审计追踪、跨院区数据同步
- 输出支持 Markdown / JSON / 飞书文档 / GitHub Issue 多格式一键导出
- 与 Obsidian 个人知识库深度集成，生成文档自动归档

**技术亮点**：
- 基于真实大模型（Kimi K2.5）生成，非模板填空
- 智能行业检测：通过关键词语义分析自动识别行业类型
- 医疗模板库已内置"病理切片借阅平台"、"医疗管理后台"等场景模板

### 2.2 Multi-Agent 协作系统

**功能描述**：
- 5 个 AI Agent 协同工作，模拟真实产品团队的决策过程
  - **CEO**：产品战略、需求分析
  - **Designer**：UI/UX 设计建议
  - **Eng Manager**：技术架构评估
  - **QA Engineer**：测试策略、安全审计
  - **Orchestrator**：任务协调、冲突解决
- Agent 状态实时可视化（idle / working / completed / error）
- 决策日志透明化：记录每个 Agent 的推理过程、置信度评分
- 自动冲突检测：当 Agent 间出现分歧时，系统自动标记并提示

**业务价值**：
让一个人也能拥有"完整产品团队"的思考能力，避免个人认知盲区。

### 2.3 可视化工作流编排

**功能描述**：
- 基于 React Flow 的拖拽式工作流画布
- 预置模板："从 0 开发新功能"、"安全审查流程"、"PRD 完整输出包"
- Agent 节点实时状态 + 整体进度条可视化
- 一键运行/停止工作流

### 2.4 实时协作与评审

**功能描述**：
- WebSocket 实时同步编辑（光标位置、选择范围）
- 评论系统：@提及、评论回复、解决状态跟踪
- 段落级锁定，避免编辑冲突
- 评审材料自动生成：议程、Q&A 预案、风险分析、决策点梳理

### 2.5 代码生成（PRD → 代码）

**功能描述**：
- **原型生成**：从 PRD 提取 UI 需求，生成 HTML + Tailwind CSS 可运行原型
- **API 生成**：生成 OpenAPI 3.0 规范和 FastAPI CRUD 代码
- **组件生成**：识别 UI 组件需求，生成 React + TypeScript 组件
- 支持 HTML / ZIP / JSON 多格式导出

**业务价值**：打通"需求文档"到"开发落地"的最后一公里，减少需求传递中的信息损耗。

### 2.6 RAG 知识库检索

**功能描述**：
- 对接 Obsidian 个人知识库，实现向量检索
- 在生成 PRD 或回答问题时，自动检索历史文档和经验沉淀
- 支持语义搜索，不只是关键词匹配

## 三、技术优势

### 3.1 技术栈（现代、主流、可扩展）

| 层级 | 技术 | 选择理由 |
|:-----|:-----|:---------|
| 前端框架 | Next.js 15 + React 19 + TypeScript | 行业主流，SEO 友好，SSR/SSG 支持 |
| 样式方案 | Tailwind CSS 4 | 开发效率高，设计系统一致性好 |
| 状态管理 | Zustand | 轻量、TypeScript 友好 |
| 工作流画布 | React Flow | 可视化编排的行业标准方案 |
| 后端框架 | FastAPI + Python 3.11 | 异步性能强，API 文档自动生成 |
| 数据库 | PostgreSQL + Redis | 生产级组合，支持高并发 |
| AI 引擎 | Kimi + Claude + OpenAI 多 Provider | 自动降级，避免单点依赖 |
| 部署方案 | Vercel (前端) + Railway (后端) | 现代化 DevOps，一键部署 |

### 3.2 工程质量

- **后端单元测试**：301 个测试用例，**全部通过**
- **E2E 全流程测试**：31 个端到端测试，**全部通过**
- **测试代码量**：约 4,786 行
- **编译零报错**：前端 `next build` 一次编译成功
- **真实 LLM 接入**：已接入 Kimi 真实 API，非 Mock 演示

### 3.3 架构设计亮点

1. **多 LLM Provider 自动降级**：Kimi 不可用时自动切换到 Claude 或 OpenAI
2. **可选认证架构**：已支持单用户免登录模式（本地自用），也保留了企业级 JWT 认证扩展能力
3. **标准化响应格式**：所有 API 统一返回 `{success, data, error, meta}` 结构，前端对接成本低
4. **技能系统可插拔**：新功能以"技能"形式注册，不侵入核心代码
5. **Obsidian 双向集成**：既是输出端（保存 PRD），也是输入端（RAG 检索知识库）

## 四、竞品对比与差异化优势

### 4.1 市场格局

| 竞品 | 定位 | 关键缺陷 |
|:-----|:-----|:---------|
| Notion | All-in-One 工作空间 | 国内访问不稳定，AI 仅限商业版，PRD 管理需自建 |
| 飞书项目 | 企业级项目管理 | 对非飞书用户门槛高，小团队功能过重，无 AI PRD 生成 |
| Jira | 敏捷开发标准 | 学习曲线陡峭，产品经理使用体验差，国内云版慢 |
| Linear | 现代 Issue 追踪 | 需翻墙，功能偏研发，不适合产品需求管理 |

### 4.2 Jarvis PM 的差异化

| 维度 | 竞品 | Jarvis PM |
|:-----|:-----|:----------|
| **AI 深度** | AI 是附加功能 | **AI 是核心引擎**，从底层重构工作流 |
| **PRD 生成** | 无或模板填空 | **对话式生成完整 PRD**，8 章标准结构 |
| **行业适配** | 通用型 | **医疗行业深度定制**，合规条款自动生成 |
| **个人知识库** | 无 | **Obsidian 深度集成**，沉淀可复用 |
| **需求到代码** | 断链 | **PRD → 原型/API/组件**，链路完整 |
| **多 Agent 协作** | 无 | **5 个 AI Agent 模拟真实团队决策** |

## 六、发展方向与规划

### 6.1 近期（1-3 个月）

1. **个人效率优化**
   - 更多行业模板（金融、教育、SaaS）
   - PRD 生成结果缓存，提升响应速度
   - 浏览器插件：一键从网页提取需求并生成 PRD

2. **医疗行业深耕**
   - 对接 HIS/医保/支付等系统接口文档，生成集成方案
   - 等保三级检查清单自动化评分
   - 多院区政策差异知识库建设

### 6.2 中期（3-6 个月）

1. **团队协作版**
   - 恢复并增强企业级权限管理
   - 团队模板库共享
   - 评审会议实时协作增强

2. **数据智能**
   - 基于历史 PRD 的智能推荐（相似项目自动参考）
   - 项目风险预测（基于进度和需求的偏离度分析）

3. **代码链路打通**
   - PRD 直接生成可运行的前端页面

   - 设计稿（Figma）与 PRD 的双向同步

### 6.3 长期（6-12 个月）

   - **团队版**：按人头订阅，支持协作和权限
   - **企业版**：私有化部署 + 行业定制 + API 对接

2. **生态建设**
   - 开放 Agent 和技能插件市场
   - 与主流工具生态集成（飞书、钉钉、企业微信、Jira）
   - 建立产品经理 AI 方法论社区
