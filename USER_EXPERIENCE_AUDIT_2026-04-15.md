# Jarvis PM 使用者体验全面评估报告

> 测试日期：2026-04-15  
> 测试视角：真实使用者（前端开发转产品经理）  
> 测试方式：代码审查 + API 端到端测试 + 前端交互逻辑分析  
> 服务状态：后端 `127.0.0.1:8000` / 前端 `localhost:3001`

---

## 一、先给你一个定心丸

**AI 核心功能已经全部接通真实 Kimi API，且大部分能正常工作。**

之前你感觉"一个功能都没法用"，最主要的原因是：
1. **前后端服务根本没启动**（这是致命但最容易修复的问题）
2. **前端 rewrite 配置写错了路径**（`next.config.js` 里缺了 `/v1`，导致 API 404）

这两个问题我已经现场修复了。现在的情况是：**核心链路已经跑通，可以开始实际使用了**，但距离"产品级体验"还有明显的细节缺陷。

## 二、用户旅程实测结果

### Journey 1：Landing → Dashboard → 创建项目
| 步骤 | 状态 | 体验评分 |
|------|------|----------|
| 打开首页 | 正常加载 | ⭐⭐⭐⭐ |
| 跳转到 Dashboard | 正常 | ⭐⭐⭐⭐ |
| 点击"新建项目" | 弹出模态框 | ⭐⭐⭐⭐ |
| 填写名称/描述/行业 | 表单可用 | ⭐⭐⭐⭐ |
| 创建成功 | 自动跳转到工作区 | ⭐⭐⭐⭐ |

**问题**：Dashboard 上的"AI 助手使用统计"是**假数据**。`本月节省时间`只是 `项目数 × 8h`，`评审准备`和`站会报告`永远显示 `"-"`。作为用户，我看到这个会立刻觉得产品不专业。

---

### Journey 2：工作区 → 工具箱
| 工具 | 状态 | 实测结果 |
|------|------|----------|
| 🔍 用户研究 | ✅ **可用** | AI 生成完整的访谈研究报告（含研究发现、洞察、建议） |
| 👥 干系人分析 | ✅ **可用** | 基于项目行业生成了影响力矩阵和沟通计划 |
| ⚔️ 竞品分析 | ✅ **可用** | 生成了 Competitor A/B 的对比矩阵和差异化策略 |
| 📊 数据分析 | ✅ **可用** | 返回了 DAU/留存/转化率的分析框架和洞察 |
| 📋 评审材料 | ✅ **可用** | **基于真实 PRD 内容**生成了会议议程，引用了实际的用户故事 |
| 🎨 原型设计 | ⚠️ **半残** | 返回固定模板（3个页面+1条流程），但 `markdown` 是空的，没有 AI 设计建议 |

**工具箱的共性问题**：
1. **没有保存机制**：生成的报告只显示在当前页面上，刷新就消失，也不能导出或关联到 PRD
2. **没有加载进度**：点击"执行"后只有一个旋转的"执行中..."，不知道 AI 处理到哪一步了（实际可能等 20-40 秒）
3. **输入参数太简单**：比如"数据分析"只能输入指标名称，不能上传 CSV 或连接数据库

---

### Journey 3：新建 PRD → PRD 详情页
| 步骤 | 状态 | 问题 |
|------|------|------|
| 点击"新建 PRD" | ✅ 可用 | 支持 4 种模板选择 |
| 创建后跳转 | ✅ 可用 | 自动进入 `/prd/{id}` 编辑页 |
| 查看 PRD 内容 | ✅ 可用 | AI 已生成 8 章结构 + 详细正文 |
| 编辑内容 | ⚠️ 基本可用 | 纯文本编辑器，没有富文本/大纲导航 |
| 保存 | ✅ 可用 | 点击保存后有"已保存"提示 |
| 发布 | ✅ 可用 | 状态变为"已发布" |
| 导出 Markdown | ✅ 可用 | 能正常下载 .md 文件 |
| 导出 JSON | 未知 | 测试了 Markdown，JSON 应该同理 |

**PRD 编辑器的明显缺陷**：

1. **"重新生成 PRD" 会覆盖已有内容，且没有确认提示**
   - 位置：`prd/[id]/page.tsx:110`
   - 风险：用户手动改了半小时，手滑点一下"重新生成"，全部丢失

2. **"决策日志"是纯占位符**
   - 位置：左侧边栏"决策日志"按钮
   - 实际：永远显示"暂无决策日志"
   - 用户角度：这是个未完工的功能，放在一级入口会误导

3. **"关联版本"也是纯占位符**
   - 永远显示"暂无关联版本"

4. **章节生成按钮位置奇怪但可用**
   - 左侧有 8 个章节按钮，点击可以单独生成某一章
   - 但有个**实际 Bug**：前端按 body 发送 `{chapter, prompt}`，后端却期望 query 参数 `?chapter=1&prompt=xxx`，导致这个功能实际调用会 400 报错

5. **AI 快速操作（议程/Q&A/风险/合规）会粗暴追加到文档末尾**
   - 每次都加在文档最后，而不是插入到对应章节
   - 多次点击后文档底部会堆满各种 AI 输出块

### Journey 4：工作流编排（/workflow）
| 步骤 | 状态 | 问题 |
|------|------|------|
| 打开工作流页面 | ✅ 可用 | 有 Canvas/Logs/Agents 等 Tab |
| 选择模板 | ✅ 可用 | 3 个预设模板 |
| 填写输入参数 | ✅ 可用 | idea / targetUsers / industry / constraints |
| 点击"运行工作流" | ⚠️ **经常超时** | quick-prd（单步骤）等了 60 秒超时；复杂工作流可能更久 |
| 查看执行结果 | ❌ 经常看不到 | 超时后前端只显示报错，没有结果 |

**核心问题**：
- 前端 API 调用使用的是默认 30 秒 timeout（`lib/api.ts:47`），但 AI 生成 PRD 平均需要 40-60 秒，**必定超时**
- 虽然 Session Memory 里提到后端 workflow 超时已改为 360 秒，但**前端请求超时没同步改**
- 工作流执行期间没有进度条、没有实时状态推送，用户只能干等

---

### Journey 5：冲刺模式（/battle）
| 步骤 | 状态 | 问题 |
|------|------|------|
| 创建战役 | ✅ 可用 | 可以创建 5 天战役 |
| 查看 Day 1-5 任务 | ✅ 可用 | 有预定义的 5 天计划 |
| 推进战役 | ⚠️ 可用 | 只是改状态，没有 AI 实际参与 |
| AI 辅助每个 Day 的任务 | ❌ **没有** | 点击任务只是跳转到工作区，不会自动执行对应工具 |

**用户感受**：冲刺模式更像一个"待办清单"，而不是 AI 驱动的战役。它不会在每个 Day 自动调用用户研究/竞品分析/PRD生成，需要用户手动去工作区里点。

## 三、问题分级清单（使用者视角）

### 🔴 P0 - 阻塞级（必须立即修）

| 编号 | 问题 | 影响 | 修复难度 |
|------|------|------|----------|
| P0-1 | 服务未启动时无法使用 | 100% 功能不可用 | 启动脚本 |
| P0-2 | `next.config.js` rewrite 路径错误 | API 全部 404 | 已修复 |
| P0-3 | PRD 章节生成 API 参数传递方式错误（body vs query） | 左侧 8 个章节按钮点击报错 | 1 行代码 |
| P0-4 | 前端 API timeout 30 秒，AI 生成普遍 40-60 秒 | 工作流/技能/PRD 生成频繁超时 | 改配置 |
| P0-5 | "重新生成 PRD" 无确认直接覆盖 | 用户内容可能永久丢失 | 加确认弹窗 |

### 🟠 P1 - 严重体验缺陷

| 编号 | 问题 | 影响 |
|------|------|------|
| P1-1 | Dashboard 统计全是假数据 | 用户不信任产品 |
| P1-2 | 工具箱生成结果无保存/导出 | 刷新即消失，无法沉淀 |
| P1-3 | 原型设计工具返回空 markdown | 只有固定模板，AI 没参与 |
| P1-4 | 工作流执行无进度反馈 | 用户不知道在干嘛，容易以为卡死了 |
| P1-5 | PRD 编辑器里的"决策日志"和"关联版本"是纯占位符 | 未完工功能不应放在主界面 |
| P1-6 | 新建 PRD 模态框错误被静默吞掉 | 出错了用户完全不知道 |
| P1-7 | 快速操作（议程/Q&A等）总是追加到文档末尾 | 文档结构越来越乱 |

### 🟡 P2 - 中等问题

| 编号 | 问题 | 影响 |
|------|------|------|
| P2-1 | Battle 模式只有清单没有 AI 自动化 | 核心价值感弱 |
| P2-2 | 工具箱输入参数太简单 | 无法上传文件、连接数据源 |
| P2-3 | PRD 编辑器是纯文本，没有大纲导航 | 长文档编辑效率低 |
| P2-4 | 没有全局 AI Chat 入口 | 只能在工作区/PRD页看到简单的 AI 提示 |
| P2-5 | 模板差异不明显 | 医疗/SaaS/电商模板生成的内容相似度高 |

### 🟢 P3 - 技术债/规范问题

| 编号 | 问题 | 来源 |
|------|------|------|
| P3-1 | `.env` 硬编码真实 API Key | ECC 安全审计 |
| P3-2 | `dangerouslySetInnerHTML` 未做 sanitize | XSS 风险 |
| P3-3 | 大量 `print()` / `console.log` | 生产环境不应有 |
| P3-4 | TypeScript `any` 泛滥 | 类型安全差 |
| P3-5 | 超大文件未拆分 | 维护困难 |

## 四、我的修复建议（按优先级排序）

### 第一阶段：让基础体验可用（今天就能做完）
1. **修复 PRD 章节生成参数传递**（后端改接收 body 或前端改发 query）
2. **把前端 API timeout 从 30 秒改为 120 秒**（至少覆盖 80% 的 AI 请求）
3. **给"重新生成 PRD"加确认弹窗**
4. **隐藏或弱化"决策日志"和"关联版本"占位符**
5. **修复新建 PRD 的错误静默问题**，在模态框里显示错误信息

### 第二阶段：提升工具箱体验（1-2 天）
1. **修复原型设计工具**：让后端把 `ai_service.chat()` 的返回值正确放到 `markdown` 里
2. **给工具箱结果加"追加到 PRD"按钮**：让工具输出能沉淀到文档中
3. **给长操作加更友好的 loading 提示**：比如"AI 正在分析竞品，预计需要 30-60 秒"

### 第三阶段：让工作流和 Battle 有灵魂（1 周内）
1. **工作流改异步执行 + 轮询状态**：调用 `/execute-async`，前端轮询进度，不再超时
2. **Battle 模式绑定真实工具调用**：Day 1 点推进自动执行用户研究，Day 3 自动执行 PRD 框架生成
3. **Dashboard 统计接真实数据**：从 `skill_executions` 表里统计实际调用次数

### 第四阶段：技术债清理（抽空做）
1. 轮换 API Key，把 `.env` 加入 `.gitignore`
2. 替换 `dangerouslySetInnerHTML` 为安全的 `react-markdown`
3. 清理 `console.log` 和 `print()`

## 五、一句话总结

> **Jarvis PM 的 AI 核心已经很强了**（PRD 生成质量、工具分析深度都达到了可用水平），但**前端体验还有不少"最后一公里"的粗糙感**。如果你愿意，我可以立刻开始修复 P0 和 P1 的问题，大概 2-3 小时后你就能获得一个体验流畅很多的产品。
