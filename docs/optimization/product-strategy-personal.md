# Jarvis PM 产品策略（个人版）

> 先自用，再分享，最后才考虑商业化
> 版本: v1.0 | 日期: 2026-04-10

---

## 核心理念

### 为什么不立即商业化？

```
❌ 错误路径:
想法 → 开发MVP → 找用户 → 发现不匹配 → 改产品 → 再试 → 失败

✅ 正确路径:
想法 → 自用验证 → 打磨到满意 → 分享给小圈子 → 自然增长 → 商业化
```

### 自用优先的优势

| 优势 | 说明 |
|:-----|:-----|
| **真实需求** | 自己天天用，痛点最真实 |
| **快速迭代** | 发现问题立即改，无需等用户反馈 |
| **深度打磨** | 不赶时间，把体验做到极致 |
| **自然口碑** | 产品够好，朋友会主动问 |
| **低成本** | 无需获客成本、运营人员 |
| **风险可控** | 失败也只是浪费自己时间 |

## 三阶段规划

### Phase 1: 自用打磨期（现在 - 3个月后）

**目标**: 让系统成为自己离不开的工具

**使用场景**:
- 日常工作中的 PRD 撰写
- 医疗信息化项目的需求分析
- 竞品分析报告生成
- 站会报告自动生成

**成功标准**:
- [ ] 每周至少用 3 次
- [ ] 比手动写 PRD 快 5 倍以上
- [ ] 输出质量达到可交付标准
- [ ] 有 3 个以上朋友主动询问

**不做的事情**:
- ❌ 不开发用户系统（只有你自己用）
- ❌ 不做多租户
- ❌ 不做付费功能
- ❌ 不做官网/landing page
- ❌ 不做推广

---

### Phase 2: 小圈子分享期（3-6个月后）

**触发条件**: Phase 1 成功标准全部达成

**目标**: 让 10-20 个朋友用起来，收集真实反馈

**分享对象**:
- 前同事（产品经理）
- 技术转型产品的朋友
- 医疗信息化行业从业者
- 独立开发者朋友

**分享方式**:
- 私聊推荐
- 小群分享
- 朋友圈展示（只展示成果，不推广）

**成功标准**:
- [ ] 5+ 人持续使用（每周至少1次）
- [ ] 收到 10+ 条改进建议
- [ ] 解决 3+ 个自己没想到的痛点
- [ ] 有人主动推荐给其他人

**不做的事情**:
- ❌ 不做公开注册
- ❌ 不做收费
- ❌ 不做客服
- ❌ 不做运营活动

---

### Phase 3: 自然增长期（6-12个月后）

**触发条件**: Phase 2 有人主动推荐，自然增长开始

**目标**: 让更多人用上，验证产品价值

**可能的路径**:
1. **开源** - 代码开源，社区共建
2. **免费工具** - 公开使用，积累口碑
3. **小范围收费** - 针对重度用户的高级功能

**决策依据**:
```
如果:
- 用户自然增长 > 50人/月
- 有人愿意付费
- 运行成本可承受
→ 考虑轻度商业化

如果:
- 增长缓慢但用户粘性高
- 社区贡献活跃
→ 考虑开源

如果:
- 用户反馈一般
- 改进动力不足
→ 保持个人工具，不商业化
```

## 功能优先级（个人版）

### P0 - 必须有的（1个月内）

| 功能 | 为什么重要 | 验收标准 |
|:-----|:-----------|:---------|
| **PRD 生成** | 核心功能，每天用 | 10分钟出初稿，可交付 |
| **医疗模板** | 贴合你的专业领域 | 切片借阅、病案复印等模板可用 |
| **本地存储** | 数据安全，无需联网 | SQLite 即可，支持导出 |
| **Web 界面** | 方便使用 | 本地启动，浏览器访问 |

### P1 - 很重要（1-3个月）

| 功能 | 为什么重要 | 验收标准 |
|:-----|:-----------|:---------|
| **实时进度** | 了解 AI 在做什么 | Phase 1 已完成 ✅ |
| **检查点** | 控制生成过程 | Phase 1 已完成 ✅ |
| **Obsidian 集成** | 融入你的知识库 | 双向同步 |
| **版本历史** | 对比修改 | 支持回滚 |
| **导出 Markdown** | 灵活使用 | 格式标准 |

### P2 - 有更好（3-6个月）

| 功能 | 为什么重要 | 触发条件 |
|:-----|:-----------|:---------|
| **多人协作** | 团队使用 | 朋友开始使用 |
| **分享功能** | 给别人看 | 有人询问 |
| **模板市场** | 更多场景 | 使用稳定后 |
| **API 接口** | 扩展性 | 有开发者朋友使用 |

### P3 - 以后再说（6个月后）

- 用户系统
- 付费功能
- 商业化运营
- 企业版

## 技术架构（个人版）

### 核心原则: 简单、本地、可控

```
┌─────────────────────────────────────────┐
│              你的电脑                    │
│  ┌─────────────┐  ┌─────────────────┐  │
│  │  Next.js    │  │   FastAPI       │  │
│  │  (前端)     │◄─┤   (后端)        │  │
│  │             │  │                 │  │
│  │  localhost  │  │  - SQLite       │  │
│  │  :3000      │  │  - 本地文件存储  │  │
│  └─────────────┘  │  - 可选: Claude  │  │
│                   │    API           │  │
│                   └─────────────────┘  │
│                            │            │
│                   ┌────────▼────────┐  │
│                   │  Obsidian Vault │  │
│                   │  (知识库同步)    │  │
│                   └─────────────────┘  │
└─────────────────────────────────────────┘
```

### 不做的事情

- ❌ 不用云服务器（本地运行）
- ❌ 不用云数据库（SQLite）
- ❌ 不做用户认证（只有你自己）
- ❌ 不做权限管理
- ❌ 不做监控告警

## 使用场景设计

### 场景 1: 日常 PRD 撰写（最高频）

```
输入: "需要一个病理切片借阅功能"
↓
AI: 确认需求细节（检查点）
↓
AI: 生成 PRD 初稿（实时进度）
↓
你: 在编辑器里修改
↓
保存: 本地 + Obsidian
↓
导出: Markdown / 飞书文档
```

### 场景 2: 竞品分析报告

```
输入: "分析一下市面上的病理系统"
↓
AI: 搜索 + 分析（web-crawler）
↓
AI: 生成竞品对比表
↓
你: 补充行业 know-how
↓
保存到: Obsidian/竞品分析/
```

### 场景 3: 站会报告

```
输入: "生成昨天的工作报告"
↓
AI: 读取 Obsidian 昨日日志
↓
AI: 整理成站会格式
↓
导出: 飞书/钉钉消息
```

## 风险与应对

| 风险 | 可能性 | 应对策略 |
|:-----|:-------|:---------|
| **坚持不下来** | 中 | 降低使用门槛，先解决一个痛点 |
| **AI 效果不够好** | 中 | 调整 Prompt，增加模板约束 |
| **没时间维护** | 高 | 接受不完美，先能用就行 |
| **朋友不感兴趣** | 中 | 不推广，专注自用 |
| **技术债太多** | 中 | 允许重构，不追求一步到位 |
